The Force.com SDK provides a set of tools that you can utilize however you want for your application. Here is an example of a typical web application’s architecture including the interactions between components.

Web Layer

The web layer can use any MVC framework. In this example, Spring Web is the MVC framework. Controller classes call application-specific services that contain the business logic.

Since the application uses Spring Web, it makes sense to use Spring Security because of their tight integration. Spring Security also integrates with other MVC frameworks so using Spring Web is not mandatory. The Spring Security Integration uses the OAuth Connector to provide OAuth authentication to the application.

If you don’t want to use Spring Security, the OAuth connector can be integrated with any other security framework, including custom frameworks. The OAuth Connector uses the API Connector to connect to the underlying Force.com APIs needed to perform the authentication.

Business Layer

Business logic is contained in application-specific classes and services that are called from the controllers in the web layer. These services use POJOs that are annotated or configured as JPA entities.

Data Access Layer

The Force.com JPA provider implements the JPA specification and uses standard JPA techniques to persist entities defined in the business layer and retrieve them from the Force.com database. The API Connector provides the connection to the Force.com APIs needed to access the Force.com database.

A variety of environment-specific configuration options are available to connect the application with different environments during the development lifecycle. If all API access goes through the connector, you can easily port the application from one environment to another without rebuilding or reconfiguring your application. For more information about configuring connections to the Force.com database, see Force.com Database Connections.
